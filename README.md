# Maria — Personal AI Agent System

A locally-hosted, multi-agent AI orchestration system built for autonomous task execution, semantic memory, and persistent knowledge management.

---

## What It Does

Maria is a personal AI infrastructure layer — a system of cooperating agents with persistent memory, a searchable knowledge base, and autonomous scheduling. It runs entirely on local hardware, with no cloud dependency for inference.

At its core, Maria is designed around a question that autonomous vehicle testing taught me: *how do you build a system that stays reliable when inputs are unpredictable?* The answer in both domains is the same — redundant context, structured validation, and tight feedback loops.

---

## Two-Agent Design

- **Maria** — primary agent running Gemma 4 12B. Handles conversation, task execution, memory management, and autonomous heartbeat checks. Context window: 220K tokens.
- **Rex** — reasoning specialist running DeepSeek-R1 14B. Delegated for deep analysis, structured thinking, and complex inference tasks. Context window: 32K tokens.

## Memory Architecture

Maria runs a five-tier memory stack:

| Tier | System | What it stores |
|---|---|---|
| 1 | In-context | Live session — dies on session end |
| 2 | LanceDB vector (memory-relay) | Auto-captured preferences, decisions, identity markers |
| 3 | Wiki sources (memory-core) | 681 ingested .md files — books, identity files, diagrams |
| 4 | Daily logs | `memory/YYYY-MM-DD.md` — raw session events |
| 5 | MEMORY.md | Curated long-term distillation — manually verified facts |

**Dual injection:** Both memory-core (wiki chunks) and memory-relay (LanceDB vector recall) fire their `before_prompt_build` hooks on every turn, injecting relevant context before the model sees the message.

**Auto-capture:** The `agent_end` hook scans up to 3 user messages per turn and stores preferences, decisions, and identity markers to LanceDB automatically — with an injection guard that blocks prompt injection attempts before storage.

**Dreaming system:** Nightly consolidation pipeline (REM → Light → Deep) promotes high-confidence short-term memories to `MEMORY.md` and prunes stale entries. Dream diary written to `DREAMS.md`.

**Memory integrity policy:** "I remember X" means a file record exists. "It is logged" means a tool call confirmed the entry. User claims are not verification. An honest "I don't know" beats a confident wrong answer.

## Behavioral Protocols

Maria operates under a set of hardcoded behavioral rules in `AGENTS.md`:

**Commit Protocol** — once a tool call returns success, the decision is closed. Confirm in one sentence and stop. No re-evaluation, no alternatives, no explaining what the output contains. The loop-break test: if the tool succeeded, you're done.

**No Uninvited Action Declarations** — answering a question is the end of the turn. No "Next Steps" sections, no "I am initiating the following sequence." If something would be valuable, surface it as one sentence: "I could do X if you want." Then wait. The decision belongs to Data.

**Document Review Protocol** — before writing any finding, `read()` the target document and the source being audited against. Quote exact text for every specific claim. If you can't paste the line that proves it, don't write the claim.

**Multi-step Isolation** — any task with 5+ steps or 3+ file writes runs as an isolated cron session with full tool access and self-destructs after completion. Keeps the main session context clean.

**Search Before You Answer** — explicit tool call before any answer about past work, decisions, or project state. "I reviewed the files" is not a search. Offering to search is not searching.

**Memory Integrity** — never confirm what cannot be verified. Tool call first, claim second.

## Tool Reference System

Maria has access to 80+ verified tools across built-in harness tools and active plugins. Tool names are sourced directly from `workspace/openclaw-source/extensions/*/openclaw.plugin.json` manifests — not inferred.

A canonical `tool-reference.md` is auto-generated by `gen-tool-reference.py` and ingested into the wiki. When uncertain about a tool name, Maria runs `wiki_search("tool reference <keyword>")` against the ground-truth reference rather than guessing. Phantom tool names are a known LLM failure mode; the reference is the fix.

The generator reruns automatically whenever `sync-openclaw-source.sh` updates the source, keeping the reference current with the installed OpenClaw version.

## Mermaid Local Editor

A filesystem-backed diagram editor built from the Mermaid.js v10.2.4 monorepo source:

- **Backend:** Node.js server (`server.js`, zero external dependencies) on loopback
- **Filesystem API:** `GET/POST/DELETE /api/diagrams/:name` — reads and writes `workspace/diagrams/*.md` directly
- **Ingest button:** One-click `openclaw wiki ingest` — pushes edited diagrams back into the wiki corpus
- **Access:** Tailscale network, HTTPS via MagicDNS
- **Service:** `mermaid-editor.service` systemd user unit — starts on boot, restarts on failure

Maria uses the editor to create, view, and maintain architecture diagrams. All diagrams live in `workspace/diagrams/` as `.md` files with mermaid code fences, rendering natively in Obsidian.

## Autonomy Roadmap

Five items that bridge the gap between reactive assistant and fully autonomous agent, in build order:

1. **Persistent Goals** — `GOALS.md` Maria owns and maintains. Without goals the heartbeat has nothing to execute against.
2. **Heartbeat that executes** — upgrade from vitals check to work loop: read goals → pick next step → run it → log → update status.
3. **Environmental triggers** — webhooks, file watchers, sensors. Pull model (heartbeat) + push model (triggers) covers the full event space.
4. **Self-evaluation loop** — act → verify immediately → re-verify after time passes → flag drift. Closes the one-shot bet problem.
5. **Outcome judgment** — effort budget rule: if N attempts without convergence, stop and escalate. The middle gear between complete and fail.

## Key Design Decisions

**Why local-only inference?**
Privacy and latency. All data stays on-device. No API costs, no rate limits, no data leaving the machine.

**Why two agents instead of one?**
Cognitive specialization. Gemma 4 handles execution and memory tasks well at 12B scale. DeepSeek-R1 is a reasoning model optimized for structured analysis. Running both lets the system match the model to the task instead of forcing one model to do everything.

**Why dual memory?**
Vector recall is great for fuzzy, semantic retrieval ("what did we decide about X last week?"). Wiki injection is better for structured reference material (books, source maps, project docs). Neither alone covers the full retrieval surface. MEMORY.md is the curated layer above both — the distilled truth that makes searches targeted instead of blind.

**Why a dreaming system?**
Auto-captured LanceDB memories are high-volume, low-curation. MEMORY.md is low-volume, high-curation. Something needs to promote the best short-term captures to long-term storage and prune what's stale. The nightly REM → Light → Deep pipeline does that without requiring manual intervention.

**Why behavioral protocols instead of just prompting?**
Prompting says "please do X." Protocols are rules that fire regardless of context. The Commit Protocol exists because a well-prompted agent will still spiral into re-evaluation loops under competing instructions. The Document Review Protocol exists because agents hallucinate audit findings. Hard rules close gaps that soft instructions leave open.

**Why a canonical tool reference?**
LLMs invent tool names. They pattern-match to what sounds plausible and report it confidently. The fix isn't stronger prompting — it's a searchable ground-truth document generated directly from source code manifests. When Maria is uncertain about a tool name, she searches the reference instead of guessing.

**Why verify before reporting?**
Agents hallucinate. They'll claim to have read a file they haven't, written content they didn't, or remember something that was never stored. Every state claim in this system requires a tool call to verify. "I read it" is not the same as the read tool returning the content.

**Why compaction instead of just a larger context window?**
A 220K token context window sounds like enough — until you've watched a session drift after 100+ messages. Compaction keeps the active context tight and high-signal. The tradeoff is intentional: precision over raw capacity.

**Why isolated task execution?**
Long multi-step tasks pollute the main session context. Each step adds noise and the model loses precision on earlier instructions. Isolated sessions start clean, stay focused, and self-destruct after completion — keeping the main session coherent.

## What I Learned

Building Maria taught me that the hard problems in AI agent systems aren't the models — it's the **infrastructure around them**.

Getting an LLM to answer a question is easy. Getting an agent to:
- Stay coherent across 100-message sessions
- Recall relevant context from weeks ago without hallucinating
- Verify its own outputs before reporting them
- Operate autonomously without drifting from its original instructions
- Diagnose and recover from context drift and memory retrieval failures
- Stop after a successful action instead of spiraling into re-evaluation
- Know which tool to call without inventing names that sound plausible
- Distinguish between "I remember X" (file record exists) and "I think X" (inference)

...that's engineering. The same class of problems I worked on testing autonomous vehicle perception systems — where a model that works 99% of the time is not good enough, and the 1% edge cases are exactly what you're there to find.

The domain changed. The discipline didn't.
